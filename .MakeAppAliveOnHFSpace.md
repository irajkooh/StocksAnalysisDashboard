# How We Keep StocksAnalysisDashboard Alive on HuggingFace Spaces

HuggingFace free-tier Spaces go to sleep after ~15 minutes of inactivity.
Once asleep, the container is frozen — nothing inside it can wake itself up.
We solve this with a **two-layer keep-alive strategy**: one running inside the app,
one running externally on GitHub.

---

## Layer 1 — APScheduler Inside the App (`app.py` + `requirements.txt`)

### Why it alone is not enough
When HF freezes the container, the Python process is suspended. Any scheduler,
thread, or loop inside the app stops. So APScheduler cannot wake a sleeping Space —
it can only keep a **running** Space from becoming idle.

### What it does
While the app is running, APScheduler sends a self-ping every 30 minutes and a
daily warm-up at 06:00 UTC. This resets HF's inactivity timer and also serves
as a hook for future scheduled tasks (e.g. pre-warming caches before market open).

### Changes made

#### `requirements.txt` — added APScheduler
```
APScheduler>=3.10.0
```

#### `app.py` — added import
```python
import requests as _requests
```

#### `app.py` — added two new functions

```python
def _ping_self(url: str):
    """Lightweight GET to keep the HF Space from going to sleep."""
    try:
        _requests.get(url, timeout=10)
        logger.info(f"[keep-alive] pinged {url}")
    except Exception as exc:
        logger.warning(f"[keep-alive] ping failed: {exc}")


def start_keep_alive_scheduler(space_url: str):
    from apscheduler.schedulers.background import BackgroundScheduler
    from apscheduler.triggers.cron import CronTrigger
    from apscheduler.triggers.interval import IntervalTrigger

    scheduler = BackgroundScheduler(timezone="UTC", daemon=True)

    # Every 30 minutes — resets HF inactivity timer
    scheduler.add_job(
        _ping_self,
        trigger=IntervalTrigger(minutes=30),
        args=[space_url],
        id="keep_alive_30m",
        replace_existing=True,
    )

    # Daily at 06:00 UTC — warm-up before market open
    scheduler.add_job(
        _ping_self,
        trigger=CronTrigger(hour=6, minute=0),
        args=[space_url],
        id="keep_alive_daily",
        replace_existing=True,
    )

    scheduler.start()
    return scheduler
```

#### `app.py` — wired into `main()` (HF Spaces only)
```python
# Step 5 — start keep-alive scheduler (HF Spaces only)
if IS_HF_SPACE:
    hf_space_url = f"https://{HF_USER}-{HF_SPACE_NAME}.hf.space".lower()
    start_keep_alive_scheduler(hf_space_url)
```

The scheduler only starts when the `SPACE_ID` environment variable is present
(set automatically by HF). It does nothing on local dev.

## Layer 2 — GitHub Actions External Pinger (`.github/workflows/keep_alive.yml`)

### Why this is the critical layer
This runs on GitHub's infrastructure — completely outside the HF container.
Even if the Space is fully asleep, GitHub's runner does a real HTTP GET to the
Space URL, which causes HF to wake the container. This is the only mechanism
that can **wake a sleeping Space**.

### File: `.github/workflows/keep_alive.yml`

```yaml
name: Keep HF Space Alive

on:
  schedule:
    - cron: "*/30 * * * *"   # every 30 minutes — prevents sleep
    - cron: "0 9 * * 1"      # every Monday 09:00 UTC — heartbeat commit
  workflow_dispatch:           # manual trigger from GitHub UI

permissions:
  contents: write

jobs:
  ping:
    name: Ping HF Space
    runs-on: ubuntu-latest
    steps:
      - name: Wake / keep alive
        run: |
          STATUS=$(curl -o /dev/null -s -w "%{http_code}" \
            --max-time 30 --retry 3 --retry-delay 5 \
            https://irajkoohi-stocksanalysisdashboard.hf.space)
          echo "Response status: $STATUS"

  heartbeat:
    name: Weekly heartbeat commit
    runs-on: ubuntu-latest
    if: github.event.schedule == '0 9 * * 1'
    steps:
      - uses: actions/checkout@v4
      - name: Write heartbeat timestamp
        run: echo "Last heartbeat: $(date -u '+%Y-%m-%d %H:%M UTC')" > .github/heartbeat.txt
      - name: Commit and push
        run: |
          git config user.name  "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add .github/heartbeat.txt
          git commit -m "chore: weekly heartbeat $(date -u '+%Y-%m-%d')"
          git push
```

### The 60-day GitHub problem — solved by the heartbeat job
GitHub automatically **pauses scheduled workflows** if the repository has no
activity for 60 days. The `heartbeat` job runs every Monday and commits a
timestamp to `.github/heartbeat.txt`, which counts as repo activity. This
prevents GitHub from ever pausing the ping schedule.

## How the Two Layers Work Together

```
Timeline (Space is running):
  t=0        App starts → APScheduler launches
  t=30min    APScheduler pings HF Space URL  ← resets inactivity timer
  t=30min    GitHub Actions also pings       ← external confirmation
  t=60min    Both ping again
  ...        continues indefinitely

Timeline (Space somehow falls asleep):
  t=0        Space is frozen, APScheduler is suspended
  t≤30min    GitHub Actions fires → GET request to Space URL
             HF receives the request → wakes the container
  t+~60sec   App restarts → APScheduler resumes
```
